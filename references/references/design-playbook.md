# Design Playbook

## Visual Direction

Choose a concrete deck style before building:

- Product defense: clean SaaS/product style, light background, blue/green accent, real UI screenshots, module cards.
- Academic report: formal typography, restrained color, diagrams and tables, fewer decorative images.
- Business pitch: bold cover, large numbers, market/problem/solution framing, fewer dense details.
- Technical architecture: system map, sequence diagrams, API/data flow visuals, compact evidence slides.

Avoid a deck that is only text boxes. Every 2-3 slides should include one of: screenshot, diagram, data chart, generated visual, process map, table, or before/after comparison.

## Slide Patterns

Use these reusable patterns:

- Cover: full-bleed or right-side visual, big literal topic title, short subtitle, date/team.
- Agenda: 4-6 sections with numbered tiles.
- Overview: 3 cards for context, architecture, value, or workflow.
- Module result: responsibility, core functions, technical implementation, completion state.
- Workflow: left-to-right process or swimlane, with actors and states.
- Architecture: front end, API/controller, service/DAO, database, external assets.
- Data result: KPI cards plus chart; avoid chart-only pages without interpretation.
- Screenshot proof: real UI screenshot with 2-4 callouts, not a tiny unreadable capture.
- Summary: completion status, integrated flow, project value, next steps.

## Typography And Layout

- Cover title: 48-68 px.
- Section title: 40-56 px.
- Slide title: 30-40 px.
- Card heading: 20-26 px.
- Body text: 17-22 px.
- Captions/footer: 10-12 px.

Use generous margins. Align to a consistent grid. Split content into another slide before shrinking important text below readable size.

## Imagery

Prefer real assets when they prove work:

- Use browser screenshots for implemented pages, dashboards, app screens, and localhost projects.
- Use generated images for atmosphere, cover visuals, or abstract module illustrations.
- Use diagrams for relationships and sequences.
- Use native charts for numeric claims.

Generated images should support the deck, not pretend to be real product evidence.

## Chinese Deck Writing

Use short, formal phrases. Replace long paragraphs with:

- `负责模块`
- `核心功能`
- `技术实现`
- `联调结果`
- `完成状态`
- `项目价值`

Keep a consistent naming style: either `客服中心模块` everywhere or `客服模块` everywhere.
